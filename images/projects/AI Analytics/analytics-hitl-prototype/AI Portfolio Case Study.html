Act I

The Translation Problem

What we were actually solving — and what we confirmed it wasn't.

Myth: The team needed better analytics.
Truth: They had Mixpanel. They had data. They had dashboards.
What they didn't have was a reliable, governed path from intent to correct query.

Humans ask questions in outcomes. "Why are D7 numbers dropping for net_cricket among organic India users on version 129?" That's the question. Mixpanel answers in events — AppLaunch, GameExited, filtered by properties, windowed by timestamps in milliseconds, with boolean comparisons on string values. Between those two things sits a gap that looks small and is enormous.

What humans speak
  • Intent and outcomes
  • Business questions in plain language
  • Relative time ("last week", "since launch")
  • Game names as known locally ("Commando", "CS")
  • Roles ("organic users who played multiplayer")
What the schema requires
  • Exact Mixpanel event names (GameExited, AppLaunch)
  • Exact property names with correct types
  • Unix timestamps in milliseconds with UTC flag
  • Canonical game identifiers (criticalstrikenetwork, cs_v3)
  • Compound filter logic spanning multiple events
This translation was happening in one person's head. The analyst. And the analyst queue never cleared. Measured: 4+ hours average from business question to validated answer for standard queries.

Three specific breaks in the translation chain

Break PointWhat happenedWhy it was dangerous
Intent → Event "Retention" is not a Mixpanel event. It's a calculation across AppLaunch events with D0 cohort logic and D-N eligibility rules. The mapping required deep schema knowledge that lived only in the analyst's head. Wrong event = wrong data. Output looked valid — retention percentages appeared, charts rendered. Error invisible without schema knowledge.
Names → Schema The same game had three names: cs_v3 (engineering), criticalstrikenetwork (Mixpanel), Commando / Critical Strike (growth and leadership). Zero reconciliation layer. A query using any non-canonical name returned either nothing or a different game's data — with no error raised.
Assumptions → Silent Failures When required information was absent, prior AI tools filled the gap silently. A date range of 2015–2025 returned when a user typed "today only" — one documented example. The most dangerous failure class: confident wrong output. Looks right. Acts wrong. Only discoverable after a decision has been made on it.
AI is probabilistic, not deterministic. We knew this from the start. The question was never "how do we make the AI always right?" It was "how do we design a system where being wrong is always visible, always catchable, and always correctable — before it causes damage?"
Act II

The AI UX Principles

Not guidelines. Architectural constraints. Every decision derived from these.

P1 AI is probabilistic — UX must manage uncertainty, not hide it
The Principle

Never present AI output as fact.

Why It Mattered

Every AI-generated artefact is a suggestion. The moment we present it as a final answer, we transfer accountability without transferring understanding.

How We Built It

Every output is labelled AI-generated. Every gate shows its reasoning context. The system ends every session with: "please provide feedback if you require any changes."

P2 Human-in-the-loop is architecture, not a UI feature
The Principle

Validation must be structurally enforced.

Why It Mattered

A UI-level confirmation can be dismissed. A PM under deadline clicks through. The gate becomes theatre — present but not functional. Meaningful oversight requires the system to be incapable of bypassing human review.

How We Built It

Seven interrupt-after nodes in LangGraph. The pipeline cannot execute past any of them without explicit human approval — enforced at the workflow layer, not the interface layer.

The atomic HITL unit — this pattern repeats at every gate in the pipeline

How one interrupt node works

step_1

AI executes a reasoning step — event selection, query generation, analysis. Produces an output.

collect_feedback (__interrupt = after)

The pipeline stops. Execution cannot continue. The user sees the AI output and must act: approve, edit, or provide feedback.

step_3 or step_4

If approved → proceed. If feedback given → incorporate correction. Either way, the human decision has been recorded.

The key point

This is not a modal dialogue. It is a workflow constraint baked into the LangGraph backend. It cannot be skipped, dismissed, or timed out.

P3 Explainability is required for trust — not an enhancement
The Principle

Users must understand before they approve.

Why It Mattered

Approving something you don't understand is not oversight. It's compliance theatre. Gates showing queries with no explanation produce visual pattern-matching, not comprehension.

How We Built It

Every generated JQL includes inline comments explaining each clause in plain language. The Technical Query layer translates intent to structured English before any code is written.

P4 Editable AI output is mandatory — approval without edit is blind trust
The Principle

Reject read-only AI outputs entirely.

Why It Mattered

Approval without the ability to edit is not validation — it is delegation. Blind edit without explanation is not correction — it is guesswork. Both fail. Editability and explainability must be designed together.

How We Built It

Every generated artefact — TQ, JQL, Python code — is editable before execution. Inline comments are generated alongside the query, not appended after. Understanding precedes editing.

P5 Validation before execution — always
The Principle

Nothing runs until a human confirms it should.

Why It Mattered

Catching a misclassification before code generation costs one correction. Catching it at output costs a full API call, Python execution, and a re-run. Validate upstream. Always.

How We Built It

Technical Query gate: intent validated in plain English before a line of code is written. Cascading edits at any gate trigger recomputation of all downstream stages.

P6 Responsible AI — the system must never make silent assumptions
The Principle

If information is missing, stop and ask. Never guess.

Why It Mattered

The most dangerous failure mode is not an error message. It is a confident result that is wrong. Filling in missing information silently is a design failure.

How We Built It

Mandatory input constraints: time range must be explicitly defined. Unresolvable game names stop the pipeline. ask_for_missing_info node interrupts whenever required context is absent.

Known Gap

Confidence & Uncertainty Signaling

What the system handles
  • Seven structural validation gates
  • Inline explainability at every gate
  • Governed learning bank accumulating corrections
  • Cascading edits triggering full recomputation
What it doesn't yet do
  • No explicit confidence scoring per query
  • No friction calibration — novel and routine queries look identical
  • No signal that "this gate deserves more scrutiny"
  • No systematic detection of confidence drift

The system currently manages uncertainty through structure. The next version should also communicate uncertainty through signal. Structure alone asks humans to apply equal scrutiny everywhere — which, in practice, means equal scrutiny nowhere.

Act IV

The System Architecture

Seven layers. Each addressing one specific translation failure.

The full production pipeline — every node, every interrupt, every path

Reading the architecture

Purple nodes (interrupt = after)

The human gates. Count them: start, ask_for_event_verification, ask_for_TQ_verification, ask_for_missing_info, ask_for_jql_verification, handle_jql_error, ask_python_result_verification. Seven gates. None optional.

Dashed arrows

Retry and correction loops. Cascading edits are architectural, not a UI feature.

verify_info + ask_for_missing_info

The responsible AI gate: if required information is absent, the system stops and asks. Principle P6 made structural.

LAYER 1Mandatory Input ConstraintsAI cannot infer temporal intent — ambiguity is blocked, not guessed
The Problem

Prior AI tools filled missing time information silently. "Today only" defaulted to 2015–2025 with no error signal. We considered probabilistic time resolution and rejected it — probabilistic resolution of time means the system's most basic parameter is uncertain, and every downstream calculation inherits that uncertainty invisibly.

The Decision

System blocks at the start interrupt node. Time range must be explicitly defined. We chose to enforce this as a hard constraint — accepting the friction of one extra input step to eliminate an entire class of silent temporal errors. Framed as "What is the start date?" not as a validation failure.

AI UX Link

Principle P6: Responsible AI. Ambiguity in input is blocked, not resolved probabilistically. We chose friction over silent failure.

LAYER 2Semantic Game Name ResolutionAlign language before aligning data
The Problem

One game, three names: criticalstrikenetwork (Mixpanel), cs_v3 (engineering), Critical Strike / Commando (growth and leadership). The alternative we considered — strict schema input — would shift the translation problem onto the user, requiring schema knowledge before asking a question.

The Decision

We chose to resolve naming ambiguity via semantic mapping, instead of forcing strict schema input — accepting minor fuzziness to avoid silent data mismatches. Unresolvable entities stop the pipeline and surface the canonical list. We chose friction over silent mismatch.

AI UX Link

Principles P5 + P6: Validate upstream. Never assume. Naming resolved at the earliest possible point — hard stop past the canonical boundary.

LAYER 3Event Selection with Human VerificationAI suggests which event — human confirms
The Problem

Mixpanel has six core event types. Selecting the wrong one produces data that answers a different question than the one asked — with no visible error if the query executes.

The Decision

System determines which events can answer the query and presents them before anything is generated. ask_for_event_verification interrupt: user reads and confirms. Only then does the pipeline proceed.

AI UX Link

Principles P2 + P5: HITL as architecture. Validation before execution. Wrong event compounds through every downstream stage.

LAYER 4Technical Query — Intent Before CodeMake the AI state what it intends to do, in plain English, before it does it
The Problem

Natural language to code is too large a single step. A retention query misclassified as an event count executes without error and returns plausible-looking data. Only discoverable post-execution — at maximum cost.

The Decision

Before any JQL is written, the system generates a Technical Query: structured English specifying cohort, event, metric, and filter logic. User approves intent. Then and only then: code generation begins.

AI UX Link

Principles P1 + P3 + P5: Manage uncertainty. Require explainability. Validate before execution. Catching a misclassification here costs one correction; at output stage, it costs a full retry cycle.

LAYER 5JQL Generation — Constrained, Commented, EditableAI writes the query. Human reads it. Human can change it.
The Problem

Generated queries without explanation produce approval-without-understanding. Read-only outputs make users accountable for something they couldn't meaningfully review.

The Decision

Every generated JQL includes inline comments explaining each clause. JQL is constrained: Events(), filter(), map() only. Editable before execution via the ask_for_jql_verification gate. Cascading edits trigger downstream recomputation.

AI UX Link

Principles P3 + P4: Explainability enables editability. Inline comments make editing a responsible act. Approval without edit = blind trust. Edit without explanation = blind modification.

LAYER 6Python Computation — Analysis with Audit RecordExecution, validation, and an immutable record of what was shown
The Problem

The computation layer has its own silent failure class: timestamps in seconds, D-N eligibility errors, boolean comparison failures on Mixpanel's string types. Each produces plausible wrong output.

The Decision

Python layer generates main(csv_file_path) with learning bank rules injected as hard requirements. Produces two outputs: matplotlib PNG at 300 DPI (immutable audit record) + ECharts JSON (interactive frontend). Today's date injected at generation time for D-N eligibility.

AI UX Link

Principle P1: The computation layer is not exempt from probabilistic failure. The PNG is immutable — it cannot be changed by a frontend update. That is a governance decision.

LAYER 7Governed Learning Bank — The Memory LayerAI must earn the right to learn. Unverified improvement is drift.
The Problem

Automatic global learning creates a new risk: a wrong correction at scale corrupts every future query of that type. Prior tools had either no memory or unvalidated propagation.

The Decision

"Remember this" → email to analytics team + founder → manual validation → Firestore update → injected as hard requirement in future prompts. Three collections: JQL, Python, TQ corrections. Session memory separate from validated global memory.

AI UX Link

Principles P2 + P6: Every correction is a change to system behaviour — reviewed before deployment. The learning bank is a micro-version of model governance.

Where I pushed back

Chat vs. Pipeline

The PM wanted chat-first. I ran two test sessions, documented the trust failure, and presented the pipeline model as the alternative. Chat was abandoned after sprint 1 results. Invisible reasoning is not a design preference — it's a trust liability.

Optional vs. Mandatory Gates

Engineering proposed optional HITL gates for users with a track record of correct approvals. I argued that optional gates create holes in the audit trail — the entire governance value depends on completeness. Gates stayed mandatory. The discipline is structural, not behavioural.

Removing RAG

When accuracy plateaued, the instinct was to add more context to prompts. I proposed removing RAG and replacing with query-type-specific injection. RAG was retrieving wrong context — increasing hallucination specificity on schema queries. The accuracy improvement after removal justified the architectural change.
